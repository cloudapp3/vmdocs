# 転送エンジン

`engine/` パッケージが純粋な転送コアです。リスナー、プロキシループ、セッション、カウンタ、ルールライフサイクルだけを所有し、それ以外は持ちません。データベース、コントロールプレーン、UI のいずれにも依存しません。これが組み込み可能性を保っています。

## プロトコル

各ルールは `protocol` フィールドでプロトコルを 1 つ選びます。

| プロトコル | 挙動 |
| --- | --- |
| `tcp` | TCP ポート転送。 |
| `udp` | UDP ポート転送。 |
| `tcp+udp` | 1 つのルールから同じポートで TCP と UDP を両方転送します。 |
| `http` | HTTP フォワードプロキシ（プレーン + `CONNECT`）。**現在のビルドでは無効**。 |
| `https` | ドメインごとの証明書を使う TLS 終端フォワード。**現在のビルドでは無効**。 |

`http` と `https` はソース（`engine/https.go`、`engine/proxy.go`）および ACME/certstore パッケージとともに存在しますが、現在のビルドでは検証時に拒否され、証明書管理は結線されていません。これらは後で再度有効化するために予約されています。

## ルールごとの制御

各ルールはプロトコルに依存しないランタイム制御を持ちます：

- **`speed_limit`** — 接続ごとの双方向レート制限（トークンバケット）、単位はバイト/秒。`0` は無制限を意味します。これは意図的にシンプルでセッション単位の仕組みであり、共有されたグローバル帯域バケットではありません。
- **`max_conn`** — 同時接続数のキャップ。キャップに達すると、新規接続は即座に切断されます。`0` は無制限を意味します。
- **`idle_timeout`** — 接続ごとのアイドルタイムアウト（秒）。`0` はデフォルト（5 分 / 300 秒）を使うことを意味します。変更するとルールは再起動します。
- **`source_ip_mode` / `source_ips`** — `off`、IPv4/IPv6 allowlist、または denylist。リテラル IP と CIDR を指定でき、1 ルール最大 256 件です。
- **`enabled`** — 無効化されたルールは設定（およびスナップショット）には残りますが起動しません。

## 送信元 IP の許可判定

TCP は `Accept` 後、接続枠の予約やターゲットへの接続より前に socket peer を評価します。UDP はセッション作成や UDP セッション枠の消費より前に、送信元データグラムを評価します。拒否されたトラフィックは転送容量を消費しません。

有効な送信元ポリシーを変更するとルールが再起動し、既存の TCP 接続と UDP セッションが閉じられます。等価な CIDR の並べ替えだけでは再起動しません。ポリシーは socket peer のみを使用します。vmflow は転送 HTTP ヘッダーや PROXY protocol メタデータを信頼せず、上流の NAT や L4 プロキシは元のクライアントアドレスを隠す場合があります。

## トラフィックカウンタ

実行中の各ルールはリアルタイムのカウンタを報告します：

- 現在の接続数
- アップロードバイト合計
- ダウンロードバイト合計
- 送信元 IP 拒否合計（TCP 接続試行または UDP データグラム）

カウンターは `vmflow ctl stats` または TUI で確認できます。送信元ポリシーの拒否数は `vmflow_rule_source_ip_denied_total` としても公開されます。統計の永続化を有効にすると、この合計はデーモン再起動後も保持されます。UDP の接続数はセッションの近似値です。

## ルールの同値性

ルールはランタイムフィールド（プロトコル、アドレス、ポート、制限、`idle_timeout`、送信元 IP ポリシー）とメタデータ（`remark`、`revision`、タイムスタンプ）の両方を持ちます。エンジンは新しいスナップショットを稼働状態と比較する際にランタイムフィールドだけを比較するため、表面的な編集が無用な再起動を引き起こすことはありません — ただし `idle_timeout` または有効な送信元 IP ポリシーの変更は再起動を引き起こします。[ルールとライフサイクル](./rules)を参照してください。
