# プレチェック

プレチェックは設定を**適用せずに**検証します。毎回の `reload` の前に自動的に実行され、またオンデマンドで実行して設定変更を安全にチェックできます。

## 実行する

```bash
vmflow ctl precheck
```

`vmflow ctl precheck` で現在の設定を検証します。

## 検査内容

プレチェックは検出項目のリストを生成し、それぞれは**エラー**または**警告**のいずれかです。エラーはリロードをブロックし、警告はブロックしません。

| 検査 | 重要度 | 備考 |
| --- | --- | --- |
| ルールモデルの検証 | error | 不正なルールフィールド。 |
| 送信元 IP ポリシー | error | 不正なモード、ホスト名、不正または空のエントリ、空の有効リスト、または 256 件を超えるリスト。 |
| `source_ip_duplicate` | warning | 以前のエントリと同じアドレスまたは CIDR に解決されるエントリ。 |
| `source_ip_redundant` | warning | 同じリスト内のより広い CIDR にすでに包含されているエントリ。 |
| `duplicate_rule_id` | error | 同じ ID がスナップショット内に複数回出現する。 |
| リスナーの衝突 | error | 2 つのルールが同じ `listen_addr:port` を要求している。 |
| ポートのバインド可否 | error | リッスンポートを実際にバインドして利用可能かを確認する。 |
| ターゲットの DNS 解決 | error | `target_addr` が解決できなければならない。 |
| HTTPS ドメイン設定 | — | _現在のビルドでは無効_（http/https プロトコルは拒否される）。 |
| ACME HTTP-01 アドレス | — | _現在のビルドでは無効_（ACME サブシステムはオフ）。 |
| 特権ポートの使用 | warning | 特権ポート（<1024）のバインドには通常、昇格された権限が必要です。 |

## レスポンスの例

```json
{
  "config_path": "./examples/config.yaml",
  "rule_count": 1,
  "result": {
    "ok": true,
    "error_count": 0,
    "warning_count": 0,
    "checked_rules": 1,
    "checked_time_ms": 1,
    "items": []
  }
}
```

## リロードでの利用方法

`vmflow ctl reload` はまず同じ検査を実行します。`error_count > 0` の場合、リロードは却下され、実行中のルールはそのまま維持されます。これにより、設定の編集をオートメーション経由で安全にデプロイできます。壊れた設定が部分的に適用されることはありません。

::: warning ローカルポート
ポートのバインド可否プローブは、ローカルリスナーを一時的に開きます。環境がローカルソケットの作成をブロックしている場合、プレチェック（ひいてはリロード）はバインド可否を完全には検証できません。
:::
