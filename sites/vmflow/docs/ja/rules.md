# ルールとライフサイクル

*ルール* は vmflow が管理する単位です。制限とメタデータを持つ 1 つの `{protocol, listen, target}` 転送エントリです。ルールには、1 件ずつ駆動できる、あるいは desired-state スナップショット全体として駆動できるライフサイクルがあります。

## 安定した識別子

各ルールは `rule_id` を持ち、設定/スナップショット内で一意でなければなりません。この ID が、何が変更されたかを判断するために vmflow が受信ルールを実行中のルールと照合する基準です。リロード間で ID を安定に保ってください — それが差分抽出を機能させる要素です。

## 単一ルール操作

| 操作 | 効果 |
| --- | --- |
| `StartRule` | 1 つのルールを開始します（すでに実行中であってはなりません）。 |
| `RestartRule` | 実行中のルールを停止して起動し直し、新しいフィールドを取り込みます。 |
| `StopRule` | 実行中のルールを停止します。設定は保持します。 |
| `RemoveRule` | ルールを停止し、稼働セットから削除します。 |

これらは対象を絞ったローカル操作に適しています。完全な desired-state がある場合は、代わりにスナップショット適用を推奨します。

## スナップショット適用

`ApplySnapshot(rules, opts)` はルールの完全な desired セットを受け取り、実行中のものと調整します。`ReplaceAll` を指定すると、スナップショットに存在しない実行中のルールはすべて停止します。

各ルールについて、適用は 1 つのアクションを生成します：

| アクション | 意味 |
| --- | --- |
| `started` | ルールはスナップショットにあるが、実行されていなかった。 |
| `restarted` | ルールは実行されていたが、ランタイムフィールドが変更された。 |
| `stopped` | ルールは実行されていたが、スナップショットにない（`ReplaceAll` の場合）。 |
| `removed` | ルールは停止され、削除された。 |
| `unchanged` | ルールは実行されており、ランタイムフィールドは変更されなかった。 |
| `failed` | ルールを開始できなかった（例: ポートが利用不可）。 |

…そしてサマリ: `Applied`、`Stopped`、`Failed`、`Total`。

::: tip 何が「変更された」か？
ランタイムフィールド（プロトコル、リッスンアドレス/ポート、ターゲットアドレス/ポート、速度制限、最大接続数、`idle_timeout`、送信元 IP モード/有効なエントリ、enabled）だけが比較されます。`remark`、タイムスタンプ、または等価な送信元 CIDR の並べ替えだけではルールを再起動**しません**。送信元ポリシーが実際に変わるとルールが再起動し、既存の TCP 接続と UDP セッションが閉じられます。
:::

## リロード

`vmflow ctl reload` は設定ファイルをリロードし、`ReplaceAll = true` で `ApplySnapshot` を実行します。まず[プレチェック](./precheck)を実行し、エラーがあればリロードは却下され、実行中のルールはそのまま維持されます。

## 状態クエリ

| メソッド | 戻り値 |
| --- | --- |
| `RunningRules` / `RunningCount` | 現在実行中のルール / 件数。 |
| `Snapshot(id)` | 1 つのルールの稼働状態。 |
| `SnapshotAll()` | すべての実行中ルールの稼働状態。 |
| `StopAll()` | すべて停止します（例: シャットダウン時）。 |
