# Telegram ボット

vmflow は軽量でチャット駆動の確認と操作のためのオプションの Telegram ボットを実行できます。スマートフォンから作業していて、サッと確認するために SSH したくない場合に便利です。

## 有効化する

ボットは**両方の**フィールドが設定された場合にのみ起動します：

```yaml
bot_token: <your-telegram-bot-token>
bot_chat: <chat-id>
```

`bot_token` は [@BotFather](https://t.me/BotFather) から取得するトークンです。`bot_chat` はボットが応答を許可される数値のチャット ID です。`bot_token` を未設定のままにすると、単にボットは起動しません。

## 認可

ボットは `bot_chat` で設定された単一チャットにロックされています — それ以外のチャットからのメッセージやコールバッククエリは無視します。ボットはコントロール API 経由ではなく、インプロセスのエンジンと**直接**通信するため、コントロール API の認証はボットには関係ない点に注意してください。コントロール API を非ループバックで公開する場合には、依然として独自のルールがあります（[デプロイメント](./deployment)を参照）。

## コマンド

| コマンド | 効果 |
| --- | --- |
| `/start` | ヘルプテキストを表示します。 |
| `/status` | 実行中ルール数、合計接続数、アップロード/ダウンロード合計。 |
| `/rules` | 実行中ルールをテーブルで一覧表示（名前、プロトコル、リッスン→ターゲット、接続数、アップ/ダウン）。 |
| `/detail <id>` | 1 つのルールの設定とライブトラフィックを表示。 |
| `/reload` | リロードを要求。確認すると**すべてのルールを停止**します（下記の注記を参照）。 |
| `/stop <id>` | ID で単一ルールを停止します（確認を求めます）。 |
| `/start_rule <id>` | ルールの開始を要求します（下記の注記を参照）。 |

破壊的な操作は、実行される前にインラインボタンで確認を求めます。

::: warning `/reload` と `/start_rule` の制限事項
- `/reload` は設定を再読み込み**しません**。確認すると `StopAll()` を呼び出し、次のように返します：_"✅ All rules stopped. Use control API /v1/reload for full config reload."_ 設定変更を実際に適用するには、コントロール API に向けて `vmflow ctl reload`（または `POST /v1/reload`）を実行してください。
- `/start_rule` はスタブです。確認すると、コントロール API の `/v1/reload` を使うよう促すヒントで返します。これは、単一ルールの再適用にはボットが保持しない完全なルール設定が必要だからです。

一方 `/stop <id>` は完全に機能します — 指定したルールを即座に停止します。
:::

## 注記

- この連携はオプションでありベストエフォートです。本番のリロードにおいてコントロール API を置き換えるものではありません。
- ボットはデーモンプロセスを共有するため、デーモンの再起動はボットも再起動します。
