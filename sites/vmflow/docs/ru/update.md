# vmflow update

Проверяет GitHub Releases и устанавливает новейшую сборку `vmflow` с тегом на месте.

```bash
vmflow update [--check] [--version <tag>]
```

Псевдоним: `vmflow u`.

## Флаги

| Флаг | Описание |
| --- | --- |
| `--check` | Только проверить наличие обновлений; не устанавливать. |
| `--version <tag>` | Установить или проверить конкретный тег релиза (например, `v0.1.1`). |

## Что делает

1. Запрашивает у GitHub Releases API информацию о `cloudapp3/vmflow`.
2. Сравнивает последний релиз с версией работающей сборки.
3. При установке: скачивает архив для текущей ОС/архитектуры, проверяет его по `checksums.txt` с помощью SHA-256, извлекает бинарный файл и атомарно заменяет работающий бинарный файл `vmflow`.

## Примеры

Проверить, доступен ли более новый релиз, без установки:

```bash
vmflow update --check
```

Установить новейший релиз:

```bash
vmflow update
```

Установить конкретную версию:

```bash
vmflow update --version v0.1.1
```

## Замечания

- Сборка `dev` (например, собранная из исходников без release ldflags) не может самообновиться без `--version`, поскольку её текущая версия неизвестна. В этом случае используйте `vmflow update --version vX.Y.Z`.
- Проверки обновлений кэшируются на 24 часа в каталоге кэша (`~/.cache/vmflow/update-check.json` или `$XDG_CACHE_HOME/vmflow`). Используйте `--version`, чтобы обойти кэш для конкретного тега.
- **Windows**: самообновление (замена бинарного файла) не поддерживается; `--check` по-прежнему работает. Для обновления на Windows переустановите через `install.sh` или архив релиза.
- Для приватных релизов или повышения лимитов GitHub API задайте `GITHUB_TOKEN` или `GH_TOKEN`.
- Самообновление заменяет работающий бинарный файл и требует права на запись в него. При ошибке прав доступа перезапустите с соответствующими привилегиями (например, `sudo`) или исправьте права пути установки.
