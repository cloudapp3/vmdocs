# 规则与生命周期

*规则（rule）* 是 vmflow 管理的基本单元：一条带限制和元数据的 `{protocol, listen, target}` 转发条目。规则拥有生命周期，既可以逐条驱动，也可以作为一整份期望状态快照来驱动。

## 稳定的身份标识

每条规则都有一个 `rule_id`，在一份配置/快照内必须唯一。vmflow 通过这个 ID 把传入的规则与正在运行的规则匹配起来，以判断发生了什么变化。请保持 ID 在多次重载之间稳定——这正是 diff 得以工作的前提。

## 单规则操作

| 操作 | 效果 |
| --- | --- |
| `StartRule` | 启动一条规则（必须尚未运行）。 |
| `RestartRule` | 停止并重启一条运行中的规则，拾取新字段。 |
| `StopRule` | 停止一条运行中的规则；保留其配置。 |
| `RemoveRule` | 停止一条规则并将其从运行集合中移除。 |

这些适合用于有针对性的本地操作。当你拥有一份完整的期望状态时，更推荐使用快照应用。

## 快照应用

`ApplySnapshot(rules, opts)` 接收完整的期望规则集合，并将其与当前正在运行的规则进行调和。配合 `ReplaceAll` 时，任何不在快照中的运行中规则都会被停止。

对于每条规则，应用过程会产生一个动作：

| 动作 | 含义 |
| --- | --- |
| `started` | 规则在快照中，但此前未运行。 |
| `restarted` | 规则此前在运行，但其运行时字段发生了变化。 |
| `stopped` | 规则此前在运行，但不在快照中（配合 `ReplaceAll`）。 |
| `removed` | 规则被停止并移除。 |
| `unchanged` | 规则此前在运行，且其运行时字段未变化。 |
| `failed` | 规则无法启动（例如端口不可用）。 |

……以及一个汇总：`Applied`、`Stopped`、`Failed`、`Total`。

::: tip 什么算作"发生了变化"？
只有运行时字段（protocol、监听地址/端口、目标地址/端口、限速、max conn、`idle_timeout`、来源 IP 模式/有效条目、enabled）会被比较。修改 `remark`、时间戳或仅调整等价来源 CIDR 的顺序**不会**重启规则。来源策略真正变化时会重启规则，并关闭已有 TCP 连接和 UDP 会话。
:::

## 重载

`vmflow ctl reload` 会重新读取配置文件，并以 `ReplaceAll = true` 运行 `ApplySnapshot`。它会先运行[预检](./precheck)；一旦出现任何错误，重载即被拒绝，正在运行的规则保持不变。

## 状态查询

| 方法 | 返回 |
| --- | --- |
| `RunningRules` / `RunningCount` | 当前正在运行的规则 / 计数。 |
| `Snapshot(id)` | 单条规则的实时状态。 |
| `SnapshotAll()` | 所有运行中规则的实时状态。 |
| `StopAll()` | 停止一切（例如在关闭时）。 |
